# Prototype Instructions

Run the local server yourself and open the preview in the browser available to this environment. Do not give the user server-start instructions when you can run it.

Before making substantial visual changes, use the Product Design plugin's `get-context` skill when the visual source is unclear or no longer matches the current goal. When the user gives durable prototype-specific design feedback, preferences, or decisions, record them in `AGENTS.md`.

When implementing from a selected generated mock, treat that image as the source of truth for layout, component anatomy, density, spacing, color, typography, visible content, and hierarchy.

## Durable product direction

- Little Worlds now opens directly into one continuous open-plan studio miniature; the active demo has no overview, room switching, floors, roof, or Explore mode.
- Little Worlds is the canonical construction language: pale laminated hardwood, captured panels, visible functional pegs, collars, softened edges, and seams that explain assembly or movement.
- The result must feel like a museum-quality architectural miniature rather than a game level, toy product, CAD model, or generic cozy renderer.
- The proof flow is: create or select an object directly in the studio, move it with the preserved magical drag behavior, and use one constrained cinematic room camera.
- The source visual truth is `../output/concept-imagery/direction-c-peg-and-pocket/01-hero-living-room.png` with `02-object-lineup.png` as the object-family reference.
- The active studio implementation follows `/Users/pavitter/.codex/attachments/644e4f9f-7f81-4e95-85fd-273c6d8d0a82/pasted-text-1.txt`: kitchen left, living room center, bedroom right, one floor, one back wall, one left wall, and three real window openings.
- The interface is permanently in creation mode: top-left wordmark plus Undo/Redo, a dedicated icon-only camera D-pad in the top-right, and one bottom bar that switches between Generate from Photo / Describe Object / Add Object and Rotate / Duplicate / Delete / Done. Generate from Photo is the primary CTA, Describe Object is secondary, and Add Object is last and tertiary.
- Generation must never silently fall back to a generic four-leg table, recolored stock category body, or ask the user to force an AI result into the stock catalog. Every generated asset follows Evidence → ObjectDNA proposal → Held Geometry compilation → validation, and stores its immutable DNA, evidence, compiler/vocabulary versions, and validation report. Low-confidence inputs may create a grammar-constrained proxy, but that proxy must still be a unique compiled ObjectDNA asset.
- Photo and text ObjectDNA proposals must compile as a single strict construction tree: exactly one root, exactly one incoming join for every non-root part, no omitted or multiply-parented parts, and no fallback placement of disconnected geometry. Reject malformed graphs before they enter the studio.
- Photo generation targets the dominant complete foreground object nearest the image center and ignores surrounding showroom furniture, labels, papers, reflections, and decor. The Held Geometry vocabulary must represent primary silhouette honestly; circular tables use a turned-vessel drum base and round-slab top rather than rectangular slabs, ring handles, or proxy boxes.
- AI-created objects live in a dedicated Generated collection inside Add Object, separate from Ready-made pieces. A generation is placed once immediately and remains reusable from Generated with its name, source, finish, seed, and category lineage intact.
- Object generation always shows an honest estimated percentage while the AI proposal is pending, then advances through Held Geometry compilation and validation only when those pipeline milestones actually occur.
- Keep object-generation quality fixed while optimizing latency: preserve the configured model, medium reasoning, strict ObjectDNA validation, and Held Geometry compiler; cache the stable schema/prompt prefix, avoid model-emitting renderer-neutral defaults, and expose Priority processing only through `OPENAI_OBJECT_SERVICE_TIER=priority` because it carries premium API pricing.
- Generated and duplicated objects must preserve their category metadata so the rendered shape cannot change into a table during placement, duplication, undo, or redo.
- Duplicate creates a sibling variation rather than a literal clone: the category and lineage stay fixed, while silhouette, proportions, finish accents, and construction details advance to a visibly different family member.
- Preserve the existing `Interactive` object selection, spring, inertia, pickup, rotation, placement, and settling behavior. Do not rewrite the object-manipulation system for scene or UI work.
- Every visible non-structural room prop must use the existing `Interactive` system. Built-in runs may move as one coherent assembly; do not leave decorative meshes that look selectable but behave as dead static objects.
- Camera input is right-drag orbit, wheel zoom, Shift+left-drag pan, Space reset, and press-and-hold top-right directional controls, constrained to the studio's open front/right sides with damping and collision protection.
- A compact environment control cluster sits immediately left of the camera D-pad. It exposes clear/cloudy/rain presets plus a 6 AM–8 PM daylight slider; time changes the real sun angle and warmth, while weather changes exterior atmosphere and daylight strength.
- Selecting any object immediately freezes every camera input and camera-driven movement; controls resume only after the object is deselected. Camera controls remain separate from the room/object editing actions.
- The living area is the visual hero: it is roughly 15–20% more generous, the sofa is larger and brighter, the coffee table sits closer, the kitchen is compressed, and the bedroom is shifted right and kept calmer.
- The default camera is a lower, closer, longer-lens miniature view. Lighting has one clear sun direction through the enlarged center window, localized sofa/floor emphasis, softer reflected fill elsewhere, and strong contact shadows without globally heavy shadows.
- Late-afternoon sun is the scene's primary visual narrator: a readable window-shaped pool crosses the living floor and coffee table, the sofa receives the warmest rim/bounce, dust is confined to the beam, and the bedroom remains quieter. Sun and filament motion must stay peripheral rather than theatrical.
- Every lamp carries a real local light inside its `Interactive` hierarchy so illumination, reflections, and nearby shadows travel with the object. Lamp influence must be clearly readable across nearby floors, walls, and furniture, with a broad warm falloff that remains subordinate to the sun. Pickup and placement may add a brief local glow, softened shadow pool, dust lift, and settling breath, but idle scenes stay understated.
- Walls are warm, subtly varied background planes; windows are approximately 25–30% larger; the floor has visible plank/grain and roughness variation; plants have barely perceptible idle sway; the platform is a chamfered, layered display plinth.
- Rugs, carpets, and floor mats are excluded from the active project for performance. Do not render built-in carpets, list them in Ready-made, or allow the generation pipeline to reintroduce them.
- Submission art direction restores the original color-blocked Little Worlds identity without changing geometry, layout, camera, interaction, or generation: ivory architecture (`#E8DFCF`), pale cream floor/platform (`#F0E3C7`), light natural wood (`#D6A568`), sparse dark structural wood (`#8A5A38`), deep blue (`#0D4778`), coral (`#E6533F`), mustard (`#D99B18`), and fresh generated foliage green (`#58A94F`). Lighting is neutral daylight with one restrained warm sun and cool fill; avoid sepia, orange fog, muddy brown upholstery, realistic wood grain, and noisy textures.
